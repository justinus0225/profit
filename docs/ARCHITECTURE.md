# P.R.O.F.I.T.
## Predictive Routing & Orchestration Framework for Intelligent Trading

---

# 1. 시스템 개요

P.R.O.F.I.T.은 암호화폐 현물 자동매매를 위한 멀티 에이전트 시스템이다.
금융 공학 팀의 실제 조직 구조를 모델링하여, 각 에이전트가 전문 영역의 업무를 수행하고
계층적 합의 메커니즘을 통해 의사결정의 신뢰성을 확보한다.

### 핵심 설계 원칙
- **계층적 상향식(Hierarchical Bottom-Up) 아키텍처**: 논문(arXiv:2602.23330)의 3-Level 구조를 차용
- **이중화(Redundancy)**: 단일 에이전트 오판 방지를 위한 교차 검증 체계
- **현물 거래 전용**: 선물/마진 없이 현물 매매만 수행
- **24/7 무중단 운영**: OpenClaw + Docker 기반 서버 배포
- **관리자 인터페이스**: OpenClaw 메시징 + Web UI를 통한 설정/모니터링

# 3. 에이전트 상세 역할 정의

## 3.1. 오케스트레이터 (Orchestrator)
> 금융팀의 CTO / 팀 리더 역할

**핵심 책임**: 전체 워크플로우 조율, 에이전트 간 합의 도출, 최종 의사결정 승인

| 업무 | 설명 |
|------|------|
| 워크플로우 관리 | 매매 사이클(분석→전략→검증→실행)의 파이프라인 조율 |
| 합의 조율 | 에이전트 간 산출물의 논리적 정합성 검증 (코사인 유사도 측정) |
| 충돌 해소 | 에이전트 간 의견 불일치 시 가중 투표(Weighted Voting)로 최종 판단 |
| 비상 제어 | 시스템 이상 감지 시 전체 매매 중단(Global Circuit Breaker) |
| OpenClaw 연동 | 관리자 메시지 수신 → 적절한 에이전트로 라우팅 → 응답 취합 |

**왜 필요한가**: 단일 에이전트 오판 방지. 퀀트 에이전트가 매수 시그널을 보내도,
경제 분석 에이전트의 거시 평가와 리스크 관리 에이전트의 승인이 없으면 실행되지 않는다.

## 3.2. 경제 분석 에이전트 (Macro/Micro Analyst Agent)
> 금융팀의 리서치 애널리스트 역할

**핵심 책임**: 시장 환경 분석, 코인별 펀더멘탈 평가, 투자 유니버스 관리

### 3.2.1. 거시(Macro) 분석
| 데이터 소스 | 수집 항목 | 분석 목적 |
|-------------|-----------|-----------|
| 미 연준(Fed) | 금리 결정, FOMC 의사록 | 글로벌 유동성 방향 |
| CPI/PPI 지표 | 물가 상승률 추이 | 인플레이션 환경 판단 |
| DXY (달러 인덱스) | 달러 강세/약세 | 위험자산 선호도 |
| Fear & Greed Index | 시장 심리 지수 | 극단적 공포/탐욕 감지 |
| BTC Dominance | 비트코인 시장 점유율 | 알트코인 시즌 판단 |
| 총 시가총액 | 암호화폐 전체 시총 | 시장 규모 추이 |
| 스테이블코인 유입량 | USDT/USDC 발행량 변화 | 시장 자금 유입 판단 |

### 3.2.2. 미시(Micro) 분석 - 코인별
| 데이터 소스 | 수집 항목 | 분석 목적 |
|-------------|-----------|-----------|
| CoinGecko/CMC API | 시가총액, 거래량, 유통량 | 기본 지표 |
| 토큰 언락 일정 | 물량 해제 날짜, 해제 비율 | 매도 압력 예측 |
| 온체인 데이터 | 대규모 지갑 이동, DEX 거래량 | 스마트 머니 추적 |
| GitHub 활동 | 커밋 수, 개발자 수 | 프로젝트 활성도 |
| SNS/뉴스 | 트위터, 레딧, 텔레그램 감성 | NLP 기반 Sentiment Score |
| 거래소 상장/폐지 | 상장 예정, 상폐 경고 | 유동성 변화 예측 |

### 3.2.3. 코인 선별 (Universe Screening)

코인 선별은 경제 분석 에이전트가 1차 필터링하고, 퀀트 에이전트가 2차 기술적 평가를 수행한다.

```
[1차 필터: 경제 분석 에이전트]
  │
  ├── 시가총액 기준: Top 100 (유동성 확보)
  ├── 일평균 거래량: 최소 $10M 이상
  ├── 거래소 상장: 타겟 거래소에 상장된 코인
  ├── 토큰 언락: 30일 이내 대규모(유통량 5%+) 언락 예정 코인 경고 플래그
  ├── 프로젝트 리스크: 해킹 이력, 규제 이슈, 팀 이탈 등
  └── 결과: 후보군 리스트 (Watchlist) + 코인별 펀더멘탈 스코어
         │
[2차 필터: 퀀트 에이전트]
  │
  ├── 기술적 지표 기반 진입 타이밍 평가
  ├── 변동성(Volatility) 분석
  ├── 상관계수(Correlation): BTC와의 상관성
  └── 결과: 최종 매매 대상 + 보유 기간 분류
```

### 3.2.4. 보유 기간 분류 기준

| 분류 | 기간 | 선별 기준 | 비중 |
|------|------|-----------|------|
| **단기(Swing)** | 1~7일 | 기술적 과매도, 급등 모멘텀, 이벤트 드리븐 | 전체의 20~30% |
| **중기(Position)** | 1~4주 | 추세 전환 확인, 온체인 축적 신호 | 전체의 40~50% |
| **장기(Core)** | 1개월+ | 높은 펀더멘탈 스코어, BTC/ETH 등 대형 코인 | 전체의 20~30% |

## 3.3. 퀀트 에이전트 (Quant/Algorithm Agent)
> 금융팀의 퀀트 트레이더 역할

**핵심 책임**: 수학적 모델 기반 매매 전략 수립, 시그널 생성, 백테스팅

### 3.3.1. 전략 엔진

| 전략 카테고리 | 사용 지표 | 적용 대상 |
|---------------|-----------|-----------|
| 추세 추종 (Trend Following) | MA(20/50/200), MACD, ADX | 중기/장기 보유 |
| 평균 회귀 (Mean Reversion) | RSI, 볼린저 밴드, Z-Score | 단기 스윙 |
| 모멘텀 (Momentum) | ROC, 거래량 급증 감지 | 단기 진입 |
| 브레이크아웃 (Breakout) | 채널 돌파, ATR 기반 변동성 | 추세 전환 시점 |

### 3.3.2. 시그널 생성 프로세스

```
[경제 분석 에이전트 산출물]          [시장 데이터 (실시간)]
        │                                   │
        ▼                                   ▼
┌───────────────────────────────────────────────┐
│              퀀트 에이전트 엔진                  │
│                                               │
│  1. 지표 연산 (MA, RSI, MACD, BB 등)           │
│  2. 멀티 타임프레임 분석 (1H, 4H, 1D)           │
│  3. 시그널 스코어링 (-100 ~ +100)              │
│  4. 보유 기간 결정 (단기/중기/장기)               │
│  5. 진입가/목표가/손절가 계산                     │
│  6. 포지션 사이징 제안                           │
└───────────────────┬───────────────────────────┘
                    │
                    ▼
         [매매 시그널 (Trade Signal)]
         - 코인: BTC/USDT
         - 방향: BUY
         - 스코어: +72
         - 진입가: $65,200
         - 목표가: $68,500 (+5.06%)
         - 손절가: $63,400 (-2.76%)
         - 보유 기간: 중기 (7~14일)
         - 포지션 비중: 전체 자금의 15%
```

### 3.3.3. 백테스팅 모듈

| 지표 | 설명 | 기준값 |
|------|------|--------|
| 총 수익률 | 전략의 누적 수익 | 벤치마크(BTC) 대비 초과 수익 |
| MDD | 최대 낙폭 | -20% 이하 시 전략 재검토 |
| Sharpe Ratio | 위험 대비 수익 | 1.0 이상 |
| 승률 (Win Rate) | 수익 거래 비율 | 50% 이상 |
| 손익비 (Risk/Reward) | 평균 수익 / 평균 손실 | 1.5 이상 |

### 3.3.4. 매매 트리거 발생 메커니즘

**매매 트리거는 퀀트 에이전트에서 발생**하며, 다음 두 가지 경로로 작동한다:

```
[경로 1: 정기 스캔 (Scheduled Scan)]
  Cron Job (15분/1시간/4시간 주기)
    → 퀀트 에이전트: 전체 Watchlist 지표 연산
    → 임계값 돌파 코인 감지
    → 시그널 생성
    → 오케스트레이터로 전달

[경로 2: 실시간 이벤트 (Event-Driven)]
  WebSocket 가격 스트림
    → 급등/급락 감지 (5분 내 ±3% 이상)
    → 퀀트 에이전트: 긴급 시그널 평가
    → 시그널 생성
    → 오케스트레이터로 전달
```

## 3.4. 리스크 관리 에이전트 (Risk Manager Agent) ★신규★
> 금융팀의 리스크 매니저 역할

**핵심 책임**: 자금 관리, 손실 통제, 포트폴리오 건전성 유지

**왜 별도 에이전트인가**: 매매 결정을 내리는 에이전트와 리스크를 통제하는 에이전트가 같으면
이해 충돌이 발생한다. 독립적인 리스크 관리 에이전트는 "브레이크 역할"을 수행한다.

### 3.4.1. 예수금 관리

| 항목 | 규칙 | 설명 |
|------|------|------|
| **최소 보유금 (Reserve)** | 전체 자산의 30% | 어떤 상황에서도 투자에 사용하지 않는 안전 자금 |
| **가용 투자금** | 전체 자산의 70% | 실제 매매에 사용 가능한 자금 |
| **단일 포지션 한도** | 가용 투자금의 20% | 하나의 코인에 집중 투자 방지 |
| **일일 최대 손실** | 전체 자산의 -3% | 도달 시 당일 신규 매매 중단 |
| **총 최대 손실** | 전체 자산의 -10% | 도달 시 전체 매매 중단, 관리자 알림 |

### 3.4.2. 현물 거래에서의 리스크 완화 전략

> **Q: 암호화폐 거래소에 헷징 수단이 있는가?**
>
> 현물 거래만 지원하는 환경에서는 전통적인 헷징(선물 숏, 옵션 풋) 수단이 제한된다.
> 따라서 다음과 같은 **현물 기반 리스크 완화 전략**을 적용한다:

| 전략 | 방법 | 설명 |
|------|------|------|
| **스테이블코인 비중 조절** | USDT/USDC 비중 확대 | 하락장 감지 시 스테이블코인 비중을 60~80%까지 확대 |
| **분산 투자** | 상관계수 기반 분산 | BTC 상관성이 낮은 코인 편입으로 포트폴리오 분산 |
| **분할 매수/매도** | DCA(Dollar Cost Averaging) | 한 번에 전량 매매하지 않고 3~5회 분할 |
| **손절(Stop-Loss)** | 자동 손절 주문 | 코인별 최대 허용 손실률 도달 시 자동 매도 |
| **트레일링 스탑** | 수익 보호 | 고점 대비 일정 비율 하락 시 자동 매도 |
| **현금화 기준** | 시장 공포 지수 | Fear & Greed ≤ 20 시 보유 포지션 축소 |

### 3.4.3. 리스크 관리 에이전트가 수행하는 헷징 의사결정 흐름

```
[시장 상태 평가]
  │
  ├── 경제 분석 에이전트 → 거시 환경 리포트 수신
  ├── 퀀트 에이전트 → 변동성/추세 지표 수신
  │
  ▼
[리스크 스코어 산출] (0~100, 높을수록 위험)
  │
  ├── 0~30 (Low Risk): 가용 투자금 100% 활용 가능
  ├── 31~60 (Medium Risk): 가용 투자금 70% 활용, 스테이블코인 30%
  ├── 61~80 (High Risk): 가용 투자금 40%, 기존 포지션 축소 시작
  └── 81~100 (Critical): 신규 매매 중단, 기존 포지션 단계적 청산
         │
         ▼
[오케스트레이터에게 리스크 레벨 전달]
  → 매매 실행 에이전트의 주문 한도 조정
  → 포트폴리오 관리 에이전트에 리밸런싱 요청
```

# 4. 단일 에이전트 오판 방지: 교차 검증 및 합의 메커니즘

## 4.1. 위험 분석

> **Q: 각 에이전트가 한 개씩만 존재했을 때의 위험은?**

| 위험 시나리오 | 발생 가능 상황 | 결과 |
|---------------|----------------|------|
| 경제 분석 오판 | NLP 감성 분석 오류로 긍정 편향 | 하락장에서 매수 시그널 |
| 퀀트 오판 | 과최적화된 전략, 지표 오계산 | 잘못된 진입/청산 시점 |
| 리스크 관리 미작동 | 한도 초과 미감지 | 과도한 손실 |
| 실행 오류 | 주문 중복, 잘못된 수량 | 의도치 않은 대량 매매 |

## 4.2. 다층 방어 체계 (Defense in Depth)

```
[시그널 생성]     [독립 검증]       [합의]          [실행 전 최종 검증]
     │                │              │                    │
  퀀트 Agent    경제분석 Agent    Orchestrator      Risk Manager
  시그널 생성    시장 방향성 확인   두 산출물 대조     한도/규칙 준수 확인
     │                │              │                    │
     └────────────────┴──────────────┴────────────────────┘
                                     │
                            [모두 승인 시에만 실행]
                                     │
                              Execution Agent
```

### 4.2.1. 합의 프로토콜 (Consensus Protocol)

논문(arXiv:2602.23330)의 방법론을 적용한 4단계 합의:

**Step 1 - 전략 제안 (Proposal)**
- 퀀트 에이전트가 매매 시그널을 생성하여 오케스트레이터에 제출
- 시그널에는 수치적 근거(지표값, 스코어)가 포함

**Step 2 - 독립 검증 (Independent Validation)**
- 경제 분석 에이전트: 현재 거시/미시 환경이 시그널 방향과 일치하는지 평가
- 리스크 관리 에이전트: 포지션 한도, 가용 자금, 리스크 레벨 준수 여부 확인
- 두 에이전트는 서로의 판단에 영향받지 않도록 **독립적으로** 평가

**Step 3 - 정합성 검증 (Reconciliation)**
- 오케스트레이터가 세 에이전트의 산출물을 수집
- 퀀트 시그널의 방향성 vs 경제 분석의 시장 평가 간 **코사인 유사도** 측정
- 유사도 임계값(예: 0.6 이상) 미달 시 시그널 기각 또는 스케일 다운

**Step 4 - 최종 승인 및 실행 (Approval & Execution)**
- 3개 에이전트 중 2개 이상이 승인해야 실행 (2-out-of-3 Quorum)
- 리스크 관리 에이전트는 **거부권(Veto)** 보유: 리스크 기준 위반 시 단독 기각 가능
- 승인된 매매 지시는 포트폴리오 관리 에이전트를 통해 매매 실행 에이전트로 전달

### 4.2.2. 에이전트 간 견제 구조

```
┌─────────────────────────────────────────────────┐
│              견제 및 균형 구조                      │
│                                                  │
│  퀀트 Agent ←── 검증 ──→ QA Agent               │
│       │                      │                   │
│       │ 시그널               │ 로직 검증          │
│       ▼                      ▼                   │
│  Orchestrator ◄── 거부권 ── Risk Manager         │
│       │                      │                   │
│       │ 승인                 │ 한도 확인          │
│       ▼                      ▼                   │
│  Portfolio Manager ──────→ Execution Agent       │
│                                                  │
│  경제분석 Agent: 독립적 시장 평가로 교차 검증       │
└─────────────────────────────────────────────────┘
```

# 10. LLM 프로바이더 추상화 계층

## 10.1. 설계 원칙

LLM은 시스템의 핵심 인프라로, 모든 에이전트의 분석/판단/자연어 처리에 사용된다.
**Claude**와 **Gemini**를 주요 프로바이더로 지원하며, OpenAI 등 다른 LLM도 확장 가능하다.
Admin UI 또는 OpenClaw 메시지를 통해 실시간으로 프로바이더를 전환할 수 있다.

| 원칙 | 설명 |
|------|------|
| **프로바이더 무관 인터페이스** | 모든 에이전트는 `LLMClient` 추상 인터페이스만 사용. 프로바이더 교체 시 에이전트 코드 수정 불필요 |
| **에이전트별 독립 설정** | 각 에이전트가 서로 다른 프로바이더/모델을 사용 가능 (예: 분석은 Claude Opus, 실행은 Gemini Flash) |
| **실시간 전환** | 프로바이더 변경 시 다음 LLM 호출부터 즉시 적용. 진행 중인 요청은 기존 프로바이더로 완료 |
| **자동 폴백** | 주 프로바이더 장애 시 자동으로 폴백 프로바이더로 전환 |
| **비용/성능 최적화** | 작업 복잡도에 따라 모델 등급 자동 선택 (고급 분석 → Opus, 단순 분류 → Haiku/Flash) |

```
     │   [동일 프로바이더 다른 모델 (예: Claude Haiku)]
     │        │
     │        ├── 성공 → 응답 반환 + 품질 저하 알림
     │        │
     │        ├── 실패
     │        │     │
     │        │     ▼
     │        │  [폴백 프로바이더 (예: Gemini Pro)]
     │        │     │
     │        │     ├── 성공 → 응답 반환 + 프로바이더 전환 알림
     │        │     │
     │        │     └── 실패 → 에이전트 일시 중단 + 관리자 알림
     │        │
     └────────┘
```

| 폴백 설정 | 기본값 | 설명 |
|-----------|--------|------|
| 최대 재시도 | 3회 | 동일 프로바이더 내 재시도 |
| 재시도 간격 | 지수 백오프 (1s, 2s, 4s) | 재시도 사이 대기 시간 |
| 프로바이더 장애 판정 | 연속 5회 실패 | 폴백 프로바이더로 자동 전환 |

## 10.7. LLM 컨텍스트 윈도우 & 에이전트 메모리 관리

### 메모리 계층 아키텍처

```
┌─────────────────────────────────────────────────────────────────┐
│                    에이전트 LLM 프롬프트 조합                      │
│                                                                   │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  [1] 시스템 프롬프트 (System Prompt)                        │  │
│  │      에이전트 역할 정의, 규칙, 출력 형식                       │  │
│  │      크기: 고정 (에이전트당 500~2,000 토큰)                   │  │
│  └────────────────────────────────────────────────────────────┘  │
│                          +                                        │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  [2] 단기 메모리 (Short-term Memory)                        │  │
│  │      최근 의사결정 이력, 진행 중 작업 컨텍스트                   │  │
│  │      저장소: Redis (TTL: 24시간)                              │  │
│  │      크기: 가변 (최대 4,096 토큰)                             │  │
│  └────────────────────────────────────────────────────────────┘  │
│                          +                                        │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  [3] RAG 검색 결과 (Long-term Memory → Retrieval)            │  │
│  │      과거 매매 기록, 전략 성과, 시장 패턴                       │  │
│  │      저장소: TimescaleDB + pgvector (유사도 검색)              │  │
│  │      크기: Top-K 결과 (최대 2,048 토큰)                       │  │
│  └────────────────────────────────────────────────────────────┘  │
│                          +                                        │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  [4] 현재 작업 (Current Task)                                │  │
│  │      실시간 시장 데이터, 분석 요청, 에이전트 산출물               │  │
│  │      크기: 가변 (작업에 따라 다름)                              │  │
│  └────────────────────────────────────────────────────────────┘  │
│                          =                                        │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  최종 프롬프트 → LLM 호출                                     │  │
│  │  ※ 에이전트별 max_input_tokens 이내로 조합                     │  │
│  └────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 에이전트별 최대 입력 토큰 제한

| 에이전트 | max_input_tokens | 근거 |
|----------|------------------|------|
| 오케스트레이터 | 16,384 | 전체 에이전트 산출물 종합 판단 필요 |
| 경제 분석 (Macro/Micro) | 8,192 | 다수 뉴스/지표 동시 분석 |
| 퀀트 | 8,192 | 복수 전략 + 시그널 데이터 |
| 리스크 관리 | 4,096 | 포지션/한도 데이터 (구조화) |
| 포트폴리오 | 4,096 | 리밸런싱 의사결정 |
| 매매 실행 | 2,048 | 주문 파라미터 검증 (단순) |
| 감성 분석 | 2,048 | 텍스트 감성 분류 (단순) |

### 프롬프트 압축 전략 (3단계)

프롬프트 조합 결과가 `max_input_tokens`를 초과할 때 순차적으로 적용:

```
[Stage 1] 단기 메모리 트리밍
   └── 오래된 항목부터 제거 (24시간 내 최신 N개만 유지)
          │
          ▼ 여전히 초과?
[Stage 2] 요약 치환 (Summarization)
   └── 단기 메모리 전체를 경량 모델(Haiku)로 요약하여 압축
       예: 10건의 의사결정 이력 (3,200 토큰) → 요약 (400 토큰)
          │
          ▼ 여전히 초과?
[Stage 3] RAG 결과 축소
   └── Top-K 결과 수 감소 (K=5 → K=3 → K=1)
       + 검색 결과 스니펫 길이 제한
```

### RAG 파이프라인 (장기 메모리 검색)

```
[에이전트 요청]
     │
     ▼
[쿼리 생성] ─── 현재 작업에서 검색 키워드/벡터 추출
     │
     ▼
[pgvector 유사도 검색]
  TimescaleDB + pgvector 확장
  ├── 테이블: agent_memory_embeddings
  │   ├── id (UUID)
  │   ├── agent_type (VARCHAR)
  │   ├── content (TEXT)          # 원본 텍스트
  │   ├── embedding (vector(768)) # 임베딩 벡터
  │   ├── metadata (JSONB)        # 시점, 카테고리, 성과 등
  │   └── created_at (TIMESTAMPTZ)
  │
  └── 검색: SELECT * FROM agent_memory_embeddings
             WHERE agent_type = '{agent}'
             ORDER BY embedding <=> query_vector
             LIMIT {top_k};
     │
     ▼
[결과 포맷팅] ─── Top-K 결과를 프롬프트 형식으로 변환
     │
     ▼
[프롬프트 조합 엔진에 전달]
```

## 13.2. P.R.O.F.I.T.의 강점 (업계 대비 우수한 점)

1. **멀티 에이전트 합의 구조**: 단일 봇 시스템 대비 의사결정 견고성이 우수. 2-out-of-3 쿼럼 + 리스크 거부권은 학술 근거(arXiv:2602.23330)가 있는 설계
2. **펀더멘탈+기술적 2단계 선별**: 상용 봇들이 기술적 지표만 사용하는 것과 달리, 거시/미시 경제 분석을 통합한 코인 선별
3. **리스크 관리 독립성**: 매매와 리스크의 분리는 기관급 설계 원칙을 정확히 반영
4. **LLM 네이티브 아키텍처**: Claude/Gemini 멀티 프로바이더 기반 자연어 분석/판단/인터페이스 통합은 기존 퀀트 플랫폼에 없는 차세대 접근. 프로바이더 추상화로 실시간 전환 및 폴백 지원
5. **투명한 의사결정 추적**: 에이전트 판단 근거, 합의 과정, 코사인 유사도까지 기록하는 매매일지

## 13.3. 식별된 핵심 갭 및 보완 사항

### 13.3.1. [P1] OMS 상태 동기화 + 멱등성 키 (Idempotency Keys)

> **위험**: 네트워크 단절 시 "좀비 주문(Zombie Order)"으로 인한 중복 주문 → 직접적 재정 손실

**현행 설계의 문제**:
매매 실행 에이전트가 주문 전송과 체결 확인에 초점이 맞춰져 있으나,
주문의 전체 생명주기(생성→전송→부분체결→완전체결→취소→수정)를 관리하는
상태 머신(Order State Machine)이 부재하다.

**보완 설계**:

```
[주문 상태 머신 (Order State Machine)]

  CREATED ──→ SUBMITTED ──→ PARTIALLY_FILLED ──→ FILLED
     │            │                │                  │
     │            ▼                ▼                  ▼
     │        REJECTED         CANCELLED          [완료]
     │            │                │
     ▼            ▼                ▼
  [에러 처리]   [로그 기록]      [잔여 수량 처리]
```

| 구현 항목 | 설명 |
|-----------|------|
| **멱등성 키(Idempotency Key)** | 각 주문에 UUID 기반 고유 키 부여. Binance `newClientOrderId`, Coinbase `client_order_id` 활용 |
| **Redis 키-상태 매핑** | 멱등성 키와 주문 상태를 Redis에 저장 (TTL: 주문 유효기간 + 여유) |
| **중복 주문 방지** | 동일 키로 재전송 시 기존 결과 반환, 새 주문 생성하지 않음 |
| **주기적 조정(Reconciliation)** | 거래소 실제 주문 상태와 내부 OMS 상태를 5분 주기로 동기화 |
| **감사 추적(Audit Trail)** | 모든 주문 이벤트(생성, 수정, 취소, 체결, 오류)를 불변 로그로 기록 |

**Execution Agent 명세 확장**:

| 기능 (기존) | 추가 기능 |
|------------|-----------|
| 주문 라우팅 | + 멱등성 키 생성 및 관리 |
| 체결 확인 | + 주문 상태 머신 전이 관리 |
| 슬리피지 모니터링 | + 거래소-내부 상태 조정(Reconciliation) |
| 서킷 브레이커 | + 좀비 주문 감지 및 자동 복구 |
| 매매 로그 기록 | + 완전한 주문 감사 추적(Audit Trail) |

### 13.3.2. [P2] 백테스트-라이브 통합 이벤트 엔진

> **위험**: 백테스팅 코드와 라이브 실행 코드가 분리되면 "백테스트에서만 잘 되는" 전략이 생산됨

**현행 설계의 문제**:
백테스팅(Backtrader/vectorbt)과 라이브 실행이 별도 코드 경로로 설계되어 있어,
Look-Ahead Bias, Survivorship Bias 위험이 존재한다.

**보완 설계: 통합 이벤트 엔진 (Unified Event Engine)**

```
┌──────────────────────────────────────────────────┐
│            Unified Event Engine                   │
│                                                   │
│  ┌──────────────┐     ┌──────────────┐           │
│  │  Historical   │     │  Live        │           │
│  │  Data Feed    │     │  Data Feed   │           │
│  │  (백테스트용)  │     │  (실시간용)   │           │
│  └──────┬───────┘     └──────┬───────┘           │
│         │                     │                   │
│         └──────────┬──────────┘                   │
│                    ▼                              │
│         ┌──────────────────┐                      │
│         │  Event Dispatcher │  ← 동일 인터페이스   │
│         │  on_tick()        │                      │
│         │  on_bar()         │                      │
│         │  on_order()       │                      │
│         │  on_fill()        │                      │
│         └────────┬─────────┘                      │
│                  ▼                                │
│         ┌──────────────────┐                      │
│         │  Strategy Code   │  ← 백테스트/라이브    │
│         │  (동일 코드)      │     동일 코드 실행    │
│         └────────┬─────────┘                      │
│                  ▼                                │
│         ┌──────────────────┐                      │
│         │  Execution Layer │                      │
│         │  - Simulated     │  ← 백테스트/Paper    │
│         │  - Live          │  ← 실제 거래소       │
│         └──────────────────┘                      │
└──────────────────────────────────────────────────┘
```

| 구현 항목 | 설명 |
|-----------|------|
| **추상 DataFeed** | Historical/Live 모두 동일 인터페이스(on_tick, on_bar)로 전략에 데이터 전달 |
| **추상 Broker** | SimulatedBroker(백테스트/Paper) / LiveBroker(실제 거래소)로 체결 분기 |
| **현실적 백테스트** | 슬리피지 모델(Volume-based), 커미션 모델, 시장 영향(Market Impact) 모델 포함 |
| **Walk-Forward Analysis** | 과최적화 방지를 위한 Walk-Forward Optimization, Monte Carlo Simulation |

### 13.3.3. [P3] CI/CD + Sandbox Forward Testing (Paper Trading)

> **위험**: 새로운 전략/코드를 프로덕션에 직접 배포하면 버그로 인한 즉각적 재정 손실 발생

**보완 설계: 4단계 전략 배포 파이프라인**

```
Stage 1: 검증 (CI)
──────────────────
  코드 변경 → Lint + Unit Test + Integration Test
  ├── 통과 → Stage 2로
  └── 실패 → QA 에이전트 피드백 → 재작업

Stage 2: 백테스트
──────────────────
  과거 6개월 데이터로 자동 백테스트
  ├── 성과 기준 충족 확인:
  │     Sharpe > 1.0
  │     MDD < -20%
  │     Win Rate > 50%
  │     손익비 > 1.5
  ├── 통과 → Stage 3로
  └── 미달 → 전략 재검토

Stage 3: Paper Trading (Forward Test)
──────────────────────────────────────
  Sandbox 컨테이너에서 실시간 데이터 + 가상 체결
  ├── 최소 2주간 운영
  ├── 실제 시장 데이터 수신, 가상 체결 엔진으로 주문 처리
  ├── 성과 리포트 자동 생성
  ├── 통과 → Stage 4로
  └── 미달 → Stage 2로 회귀

Stage 4: 프로덕션 배포
──────────────────────
  관리자 수동 승인 → 소액(전체 자금 10%) 실거래 시작
  ├── 1주간 소액 운영
  ├── 성과 확인 후 전체 자금 적용
  └── 이상 시 자동 롤백
```

**Docker 환경 추가 서비스**:

| 서비스 | 역할 |
|--------|------|
| `profit-sandbox` | Paper Trading 전용 컨테이너. 실시간 데이터 수신 + 가상 체결 |
| `profit-ci` | 코드 변경 시 자동 테스트/백테스트 실행 |

**Stage Gate 세부 통과 기준**:

| Stage | 지표 | 기준 | 측정 기간 |
|-------|------|------|-----------|
| Stage 2 (백테스트) | Sharpe Ratio | > 1.0 | 과거 1~3년 |
| | MDD (최대 낙폭) | < -20% | 과거 1~3년 |
| | Win Rate (승률) | > 50% | 과거 1~3년 |
| | Profit Factor (손익비) | > 1.5 | 과거 1~3년 |
| Stage 3 (Paper Trading) | 위 백테스트 기준 동일 충족 | - | 최소 2주 실시간 |
| | API 에러율 | < 0.1% | 연속 2주 |
| | 에이전트 응답 지연 P99 | < 500ms | 연속 2주 |
| | 메모리 누수(Memory Leak) | 미발생 | 연속 2주 |
| | 서킷 브레이커 비정상 발동 | 0회 | 연속 2주 |
| Stage 4 (부분 실거래) | 슬리피지 | 백테스트 대비 오차 < 0.3% | 최소 2주 |

| | 서킷 브레이커 발동 | 연속 0회 | 최소 2주 |

### 13.3.4. [P4] 에이전트 성과 추적 및 동적 가중치

> Numerai의 메타모델처럼, 에이전트별 과거 정확도를 기반으로 합의 가중치를 동적 조정

**현행 설계의 문제**:
"가중 투표(Weighted Voting)"가 명시되어 있으나, 가중치 산출 방법이 불명확하다.

**보완 설계**:

| 추적 지표 | 설명 | 가중치 반영 |
|-----------|------|-------------|
| 시그널 정확도 | 에이전트 예측 방향 vs 실제 가격 변동 | 최근 30일 이동 정확도 |
| 리포트 신뢰도 | 분석 리포트의 결론 vs 실제 시장 방향 | EMA(지수이동평균) 기반 |
| 리스크 판단 정확도 | 리스크 경고 발동 vs 실제 손실 발생 | 적중률 기반 |

```
[에이전트 성과 추적 흐름]

  매매 완료 → 실제 결과 기록
       │
       ▼
  각 에이전트의 과거 판단과 대조
       │
       ├── 경제 분석: 시장 방향 예측 정확도
       ├── 퀀트: 시그널 스코어 vs 실제 수익률 상관계수
       └── 리스크: 위험 경고의 적중률
       │
       ▼
  에이전트별 신뢰도 점수 갱신 (0.0 ~ 1.0)
       │
       ▼
  다음 합의 투표 시 가중치로 반영
  (신뢰도가 높은 에이전트의 의견에 더 높은 가중치)
```

### 13.3.5. [P5] 거래 비용 분석 (TCA) 모듈

> 기관 펀드 수준의 실행 품질 측정 및 개선

| TCA 항목 | 설명 |
|----------|------|
| **사전 분석 (Pre-Trade)** | 호가창 깊이, 스프레드, 예상 슬리피지를 기반으로 최적 주문 방식 결정 |
| **실시간 분석 (In-Trade)** | 체결 진행 중 슬리피지 모니터링, 임계값 초과 시 주문 일시중단 |
| **사후 분석 (Post-Trade)** | Implementation Shortfall 계산: 의사결정 시점 가격 vs 실제 체결 가격 차이 |

**실행 알고리즘 확장**:

| 알고리즘 | 사용 조건 | 설명 |
|----------|-----------|------|

### 13.3.8. [P8] 거래소 API Rate Limiting (Token Bucket)

> **위험**: 다수 에이전트의 병렬 API 호출이 거래소 Rate Limit을 초과하면 IP 차단 → 주문 실행 불가

**현행 설계의 문제**:
데이터 엔지니어(가격 스트림), 매매 실행 에이전트(주문 전송), OMS(상태 동기화), 퀀트(지표 조회) 등
다수의 에이전트가 단일 거래소 API 엔드포인트에 비동기 병렬 호출을 수행하지만,
전체 호출량을 중앙에서 조율하는 메커니즘이 부재하다.

**거래소 Rate Limit 기준** (Binance 기준):

| 항목 | 제한 | 초과 시 결과 |
|------|------|-------------|
| REST API 가중치 | 1,200 weight / 분 | HTTP 429 에러 + 일시 차단 |
| 주문 전송 | 10회 / 초, 100,000회 / 일 | HTTP 429 에러 |
| WebSocket 연결 | 5회 연결 시도 / 초 | 연결 거부 |
| IP 차단 | 반복 위반 시 | 수분~수시간 전체 차단 |

**보완 설계: 중앙 집중 Rate Limiter**

```
┌─────────────────────────────────────────────────────────┐
│              Centralized Rate Limiter                    │
│                                                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │ Execution│ │   OMS    │ │  Quant   │ │ Data Eng │   │
│  │ Agent    │ │          │ │  Agent   │ │          │   │
│  │ 우선순위:10│ │ 우선순위:5│ │ 우선순위:3│ │ 우선순위:3│   │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘   │
│       │            │            │            │          │
│       └────────────┴────────────┴────────────┘          │
│                        │                                 │
│                        ▼                                 │
│  ┌──────────────────────────────────────────────────┐   │
│  │           Priority Queue (우선순위 큐)             │   │
│  │                                                   │   │
│  │  높음: 주문 전송/취소 (Execution Agent)             │   │
│  │  중간: OMS 상태 동기화, 잔고 조회                    │   │
│  │  낮음: 시세 조회, 지표 데이터 수집                    │   │
│  └──────────────────┬───────────────────────────────┘   │
│                     │                                    │
│                     ▼                                    │
│  ┌──────────────────────────────────────────────────┐   │
│  │        Token Bucket (Redis 기반)                   │   │
│  │                                                   │   │
│  │  Key: ratelimit:{exchange}                        │   │
│  │  토큰 용량: 1,000 weight (안전 마진 200 확보)       │   │
│  │  토큰 보충: 16.67 weight/초 (1,000/60)             │   │
│  │                                                   │   │
│  │  토큰 충분 → 요청 통과, 토큰 차감                    │   │
│  │  토큰 부족 → 대기 (백프레셔) 또는 거부               │   │
│  └──────────────────┬───────────────────────────────┘   │
│                     │                                    │
│                     ▼                                    │
│              ┌──────────────┐                            │
│              │ Exchange API │                            │
│              │ (Binance 등) │                            │
│              └──────────────┘                            │
└─────────────────────────────────────────────────────────┘
```

| 구현 항목 | 설명 |
|-----------|------|
| **Redis Token Bucket** | `ratelimit:{exchange}` 키에 현재 토큰 수 저장. Lua 스크립트로 원자적 차감/보충 |
| **엔드포인트별 가중치 추적** | Binance API는 엔드포인트마다 다른 가중치 소모. 각 요청의 weight를 정확히 차감 |
| **우선순위 큐** | 주문 실행 > OMS 동기화 > 데이터 수집. 토큰 부족 시 낮은 우선순위 요청 대기 |
| **백프레셔(Backpressure)** | 토큰 부족 시 에이전트에 "N초 후 재시도" 응답 반환. 최대 대기 30초 초과 시 거부 |
| **지수 백오프(Exponential Backoff)** | HTTP 429 수신 시 1초 → 2초 → 4초 → 8초 간격으로 재시도. 최대 5회 |
| **에이전트별 할당량** | 전체 토큰 예산 중 에이전트별 최소 보장량 설정 (매매 실행: 40%, 나머지: 각 20%) |
| **메트릭 수집** | 토큰 소모율, 429 에러 빈도, 평균 대기 시간을 Prometheus 메트릭으로 수집 |

### 13.3.9. [P9] 경쟁 상태(Race Condition) 방지: 분산 락(Distributed Lock)

> **위험**: 다수 에이전트가 동일한 공유 자원(잔고, 포지션)을 동시에 읽기/쓰기하면 초과 주문 → 직접적 재정 손실

**현행 설계의 문제**:
매매 실행 에이전트가 가용 잔고를 조회(Read)한 뒤 주문을 전송(Write)하는 사이에,
다른 시그널에 의한 별도 주문이 동일한 잔고를 읽어 포지션 사이징을 계산하면
이미 할당된 금액이 중복 투입되는 초과 주문(Over-ordering)이 발생한다.

**경쟁 상태 발생 시나리오**:

```
[시간]        [에이전트 A: BTC 매수]         [에이전트 B: ETH 매수]
──────────────────────────────────────────────────────────────────
t=0ms    잔고 조회: 700,000 USDT 가용
t=1ms                                   잔고 조회: 700,000 USDT 가용
t=2ms    포지션 계산: 140,000 USDT 할당
t=3ms                                   포지션 계산: 140,000 USDT 할당
t=4ms    주문 전송: BTC 140,000 USDT
t=5ms                                   주문 전송: ETH 140,000 USDT
──────────────────────────────────────────────────────────────────
결과: 총 280,000 USDT 주문 (의도: 각각 최대 20% = 140,000)
      단일 코인 한도는 지켰으나 총 가용금 초과 위험 발생
```

**보완 설계: Redis Redlock 기반 분산 락**

```
[분산 락 적용 후]

[에이전트 A: BTC 매수]                    [에이전트 B: ETH 매수]
     │                                        │
     ▼                                        │
Lock 획득: lock:balance (TTL: 10s)           │
     │                                        ▼
잔고 조회: 700,000 USDT                   Lock 대기 (lock:balance 점유 중)
포지션 계산: 140,000 USDT                      │
잔고 차감: 700,000 → 560,000 USDT              │
주문 전송: BTC 140,000 USDT                    │
     │                                        │
Lock 해제: lock:balance                       │
     │                                        ▼
     │                                   Lock 획득: lock:balance
     │                                   잔고 조회: 560,000 USDT ← 정확한 값
     │                                   포지션 계산: 112,000 USDT
     │                                   잔고 차감: 560,000 → 448,000 USDT
     │                                   주문 전송: ETH 112,000 USDT
     │                                   Lock 해제: lock:balance
```

**공유 자원별 락 전략**:

| 공유 자원 | 락 키 | 락 범위 | TTL | 설명 |
|-----------|-------|---------|-----|------|
| 가용 잔고 | `lock:balance` | 전역 | 10초 | 잔고 조회→차감→주문을 원자적으로 수행 |
| 포지션 데이터 | `lock:position:{symbol}` | 심볼별 | 5초 | 동일 코인 동시 주문 방지 |
| OMS 주문 슬롯 | `lock:order:{symbol}` | 심볼별 | 5초 | 동일 코인 중복 주문 방지 |
| 설정값 | Optimistic Lock (버전 카운터) | 키별 | - | 동시 설정 변경 충돌 방지 |

| 구현 항목 | 설명 |
|-----------|------|
| **Redlock 알고리즘** | Redis 단일 인스턴스 기반 분산 락. `SET key value NX PX {ttl}` 원자적 획득 |
| **TTL 자동 해제** | 락 보유 에이전트 크래시 시 TTL 만료로 자동 해제 → 데드락 방지 |
| **락 재시도** | 획득 실패 시 100ms 간격으로 최대 3회 재시도 |
| **락 세분화** | 잔고(전역), 포지션/주문(심볼별)으로 구분하여 불필요한 대기 최소화 |
| **임계 섹션 정의** | 잔고 조회 → 포지션 사이징 → 주문 전송을 하나의 임계 섹션으로 묶음 |
| **Optimistic Lock** | 설정값 변경은 버전 카운터 기반. 읽기 시 버전 확인 → 쓰기 시 버전 일치하면 적용 |

### 13.3.10. [P10] 데이터 품질 / 이상치 탐지 / 데이터 힐링

> **위험**: 거래소 API의 이상 데이터(스파이크, 결측, 지연)가 그대로 지표 계산에 유입되면
> 잘못된 시그널 → 오매매로 직결된다.

**현행 설계의 문제**:
데이터 엔지니어 에이전트 역할에 "데이터 정제"가 1줄 언급되어 있으나,
이상치 탐지 기준, 힐링 방법, 격리 정책 등 구현 상세가 없다.

**보완 설계: 3단계 데이터 품질 파이프라인**

```
[거래소 WebSocket / REST 수집]
          │
          ▼
┌─────────────────────────────────────────────────────────┐
│  Stage 1: 이상치 탐지 (Anomaly Detection)                │
│                                                           │
│  ┌─────────────────────┐  ┌─────────────────────────┐   │
│  │ Z-Score 기반         │  │ IQR (사분위 범위) 기반    │   │
│  │ |z| > threshold     │  │ Q1 - 1.5×IQR 미만 or   │   │
│  │ (기본: 4.0)         │  │ Q3 + 1.5×IQR 초과       │   │
│  │                     │  │ (기본 IQR 배수: 3.0)     │   │
│  └────────┬────────────┘  └────────────┬────────────┘   │
│           └──────────┬─────────────────┘                 │
│                      ▼                                    │
│           OR 조건으로 이상치 판정                            │
└──────────────────────┬──────────────────────────────────┘
                       │
          ┌────────────┴────────────┐
          │ 정상                     │ 이상치
          ▼                         ▼
┌──────────────────┐  ┌────────────────────────────────────┐
│  Stage 3:         │  │  Stage 2: 데이터 힐링 (Data Healing) │
│  삽입 전 검증     │  │                                      │
│  (Pre-Insert      │  │  방법 (우선순위 순):                   │
│   Validation)     │  │  1. 선형 보간 (Linear Interpolation)  │
│                   │  │  2. Forward Fill (이전 값 유지)        │
│  ・ NOT NULL 확인  │  │  3. 이동 평균 대체 (MA Replacement)   │
│  ・ 타임스탬프 연속 │  │                                      │
│  ・ 가격 > 0      │  │  ※ healing_flag = true 마킹           │
│  ・ 볼륨 ≥ 0      │  │  ※ 원본은 quarantine 테이블 보관       │
└──────────────────┘  └────────────────────────────────────┘
          │                         │
          └────────────┬────────────┘
                       ▼
              [TimescaleDB 삽입]
```

**비상 중단 규칙**:
- 10분 윈도우 내 이상치 비율 > 30% → 해당 심볼 수집 자동 중단
- 관리자 알림 (OpenClaw): "⚠️ BTC/USDT 데이터 품질 경고: 이상치 43%. 수집 중단됨"
- 관리자가 수동 재개하거나, 이상치 비율이 정상화되면 자동 재개

| 구현 항목 | 설명 |
|-----------|------|
| **Z-Score 탐지** | 슬라이딩 윈도우(기본 100개) 기반 평균/표준편차 계산 |
| **IQR 탐지** | 슬라이딩 윈도우 기반 사분위 계산, 극단값 탐지에 강건 |
| **힐링 엔진** | 이상치 유형별 최적 힐링 방법 자동 선택 |
| **격리 테이블** | `data_quarantine` 테이블에 원본 이상치 + 탐지 메타데이터 보관 |
| **healing_flag** | 힐링된 데이터 포인트에 플래그 마킹, 지표 계산 시 가중치 감소 가능 |
| **비상 중단** | 이상치 비율 임계치 초과 시 자동 수집 중단 + 알림 |

### 13.3.11. [P11] LLM 컨텍스트 윈도우 & 메모리 관리

> **위험**: 에이전트가 LLM에 무제한 컨텍스트를 전송하면 토큰 비용이 폭증하고,
> 컨텍스트 윈도우 초과 시 중요 정보가 잘려 환각(Hallucination)이 발생한다.

**현행 설계의 문제**:
Section 10에서 LLM 프로바이더 추상화는 완비되었으나,
에이전트 메모리 관리(단기/장기) 및 프롬프트 조합 시 토큰 제한 전략이 없다.

**보완 설계**: → **Section 10.7 참조** (본 문서 상단)

핵심 구조 요약:

| 구성요소 | 저장소 | TTL/보존 | 용도 |
|----------|--------|----------|------|
| 시스템 프롬프트 | 코드 내장 | 영구 | 에이전트 역할 정의 |
| 단기 메모리 | Redis | 24시간 | 최근 의사결정 이력, 진행 중 작업 |
| 장기 메모리 | TimescaleDB + pgvector | 영구 | 과거 매매 기록, 전략 성과, 시장 패턴 |

| 프롬프트 압축 | Haiku 경량 모델 | - | 토큰 초과 시 3단계 압축 적용 |

### 13.3.12. [P12] 콜드 스타트 / 워밍업 시퀀스

> **위험**: 시스템 재시작 후 기술적 지표(RSI, MA 등)가 충분히 계산되지 않은 상태에서
> 매매가 시작되면 무효 시그널 기반의 오매매가 발생한다.

**현행 설계의 문제**:
TRADING_FLOW.md Section 9.1에 5단계 기본 복구만 있으며,
지표 워밍업 대기, 부분 준비 허용, trading_enabled 자동 제어가 없다.

**보완 설계: 6단계 부트 시퀀스**

```
[시스템 시작]
     │
     ▼
Phase 0: 인프라 점검 ─── Docker 컨테이너 헬스체크
     │                   TimescaleDB, Redis, PgBouncer 연결 확인
     │                   ※ 실패 시 재시도 3회 → 알림 + 중단
     ▼
Phase 1: 데이터 복구 ─── WebSocket 재연결
     │                   누락 데이터 백필 (candle_backfill_multiplier × 필요 캔들)
     │                   데이터 품질 검증 (P10 파이프라인 적용)
     ▼
Phase 2: 지표 워밍업 ─── 기술적 지표 사전 계산
     │
     │  ┌──────────────────────────────────────────────────┐
     │  │ 지표별 최소 필요 캔들 수                            │
     │  ├──────────────┬──────────────┬────────────────────┤
     │  │ 지표          │ 최소 캔들     │ 1h 캔들 기준 시간   │
     │  ├──────────────┼──────────────┼────────────────────┤
     │  │ RSI(14)      │ 14           │ 14시간              │
     │  │ MACD(12,26,9)│ 35           │ 35시간              │
     │  │ BB(20)       │ 20           │ 20시간              │
     │  │ MA(50)       │ 50           │ ~2일                │
     │  │ MA(200)      │ 200          │ ~8일                │
     │  │ ATR(14)      │ 14           │ 14시간              │
     │  │ OBV          │ 1            │ 즉시                 │
     │  └──────────────┴──────────────┴────────────────────┘
     │
     │  ※ 부분 준비 허용: RSI/MACD/BB 완료 시 해당 전략 선 활성화
     │     MA(200) 등 장기 지표는 백필 데이터로 계산
     ▼
Phase 3: OMS 동기화 ─── 거래소 미체결 주문 조회
     │                   내부 OMS 상태와 동기화 (P1 멱등성)
     │                   포지션 + 잔고 정합성 확인
     ▼
Phase 4: 헬스체크 ──── 전 에이전트 heartbeat 수신 확인
     │                  에이전트별 준비 상태 (ready/warming/error)
     │                  Prometheus 메트릭 노출 시작
     ▼
Phase 5: 매매 활성화 ── trading_enabled = true (자동 전환)
                        준비된 전략부터 순차 활성화
                        OpenClaw 알림: "✅ 시스템 부트 완료 (소요: 3m 42s)"
```

**trading_enabled 자동 제어**:
- 부트 시작: `trading_enabled = false` (자동 설정)
- Phase 4 완료 + 최소 1개 전략 준비: `trading_enabled = true`
- 모든 지표 워밍업 완료 전에는 준비된 전략만 활성화

| 구현 항목 | 설명 |
|-----------|------|
| **BootSequenceManager** | 6단계 순차 실행, 단계별 타임아웃, 상태 리포트 |
| **지표 워밍업 체커** | 지표별 최소 캔들 충족 여부 모니터링 |
| **부분 활성화** | 준비된 지표의 전략만 선 활성화, 나머지는 대기 |
| **백필 배수** | `candle_backfill_multiplier` 설정으로 필요 캔들의 N배 백필 |
| **부트 타임아웃** | 전체 부트 `warmup_timeout_minutes` 내 미완료 시 알림 |

> **TRADING_FLOW.md 교차 참조**: Section 9.1에서 이 6단계 부트 시퀀스를 상세히 기술한다.

### 13.3.13. [P13] DB 커넥션 풀링 (PgBouncer)

> **위험**: 9개 에이전트 + FastAPI + 데이터 수집기가 각각 독립적으로 TimescaleDB에
> 연결하면, 동시 연결 수가 PostgreSQL `max_connections` (기본 100)을 초과하여
> "too many connections" 에러로 시스템이 중단된다.

**현행 설계의 문제**:
Docker Compose에 TimescaleDB는 있으나 커넥션 풀링 미들웨어가 없다.
비동기 에이전트 9개가 각각 SQLAlchemy 풀을 열면 연결 수가 급증한다.

**보완 설계: PgBouncer 트랜잭션 풀링**

```
┌───────────────────────────────────────────────────────────┐
│                    에이전트 / 서비스 계층                     │
│                                                             │
│  Orchestrator  Analyst  Quant  Risk  Portfolio  Executor   │
│  DataEngineer  Developer  QA    FastAPI    DataCollector   │
│       │           │       │      │       │        │         │
│       └─────────┬─┴───┬───┴──┬───┴───┬───┘        │         │
│                 │     │      │       │             │         │
│                 ▼     ▼      ▼       ▼             ▼         │
│  ┌─────────────────────────────────────────────────────┐    │
│  │           SQLAlchemy AsyncEngine (per-agent)         │    │
│  │           pool_size=2, max_overflow=3                │    │
│  └──────────────────────┬──────────────────────────────┘    │
│                          │ (최대: 에이전트 수 × 5 = ~55)      │
└──────────────────────────┼──────────────────────────────────┘
                           │
                    ┌──────▼──────┐
                    │  PgBouncer  │
                    │  port: 6432 │
                    │             │
                    │  모드: transaction │
                    │  default_pool_size: 20   │
                    │  max_client_conn: 200    │
                    │  reserve_pool_size: 5    │
                    └──────┬──────┘
                           │ (실제 DB 연결: 20~25개)
                    ┌──────▼──────┐
                    │ TimescaleDB │
                    │ port: 5432  │
                    │             │
                    │ max_connections: 50     │
                    │ (PgBouncer가 멀티플렉싱) │
                    └─────────────┘
```

**에이전트별 커넥션 풀 차등 설정**:

| 에이전트 / 서비스 | pool_size | max_overflow | 근거 |
|-------------------|-----------|--------------|------|
| 데이터 엔지니어 | 3 | 5 | 고빈도 수집 + 벌크 INSERT |
| 퀀트 | 2 | 3 | 지표 조회 + 시그널 저장 |
| 매매 실행 | 2 | 3 | OMS 상태 업데이트 (빠른 응답 필수) |
| 기타 에이전트 | 1 | 2 | 간헐적 조회 위주 |
| FastAPI | 3 | 5 | Admin UI 요청 처리 |

| 구현 항목 | 설명 |
|-----------|------|
| **PgBouncer 서비스** | Docker Compose에 `pgbouncer` 서비스 추가 (Section 5.2) |
| **트랜잭션 풀링** | 쿼리 단위가 아닌 트랜잭션 단위로 연결 공유 (Prepared Statement 호환) |
| **에이전트별 풀** | SQLAlchemy `pool_size`, `max_overflow`를 에이전트 유형별 차등 |
| **모니터링** | PgBouncer `SHOW STATS` 메트릭을 Prometheus로 수집 |
| **PostgreSQL 최적화** | `max_connections` 50으로 축소 (PgBouncer가 멀티플렉싱) |

# 14. 자율 진화 퀀트 아키텍처 (Strategy Evolution)

기존 Phase 1~12에서 구현된 고정 전략 기반 트레이딩에 **자율 진화 메커니즘**을 추가한다.
전략이 자동으로 생성·최적화·검증·승격/강등되는 폐쇄 루프(Closed-Loop)를 구축한다.

## 14.1. 아키텍처 개요

```
┌──────────────────────────────────────────────────────────────────┐
│                 Strategy Evolution Pipeline                       │
│                                                                   │
│  ┌─────────────┐    ┌──────────────┐    ┌───────────────┐        │
│  │  Strategy    │    │  Walk-Forward│    │  Shadow       │        │
│  │  Generator   │───▶│  Optimizer   │───▶│  Tester       │        │
│  │  (LLM, opt-in)│   │  (WFO)       │    │  (가상 실행)    │        │
│  └─────────────┘    └──────────────┘    └───────┬───────┘        │
│         │                                        │                │
│         │ CANDIDATE                              │ 승격/강등       │
│         ▼                                        ▼                │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │              Strategy Registry (생명주기 관리)              │    │
│  │  CANDIDATE ──▶ SHADOW ──▶ LIVE ──▶ DEPRECATED             │    │
│  └──────────────────────────────────────────────────────────┘    │
│                              │                                    │
│                              ▼                                    │
│  ┌──────────────────────────────────────────────────────────┐    │
│  │           Regime Classifier (시장 국면 분류)                │    │
│  │  TRENDING / RANGING / VOLATILE → 전략 가중치 동적 조정      │    │
│  └──────────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────────┘
```

### 핵심 설계 원칙

| 원칙 | 설명 |
|------|------|
| **Additive 확장** | 기존 에이전트 이름/역할 변경 없이 서브모듈 추가로 확장 |
| **Zero 신규 의존성** | 기존 pandas, numpy, pandas-ta만 사용 (HMM은 optional dep) |
| **보안 우선** | LLM 전략 생성은 `generation_enabled=False` 기본값, AST 필터링 |
| **기존 인프라 재활용** | BacktestEngine, SimulatedBroker, ForwardTester 직접 재사용 |

## 14.2. Strategy Registry — 전략 생명주기 관리

모든 전략(빌트인 + LLM 생성)의 생명주기를 중앙에서 관리하는 레지스트리.

### 14.2.1. 상태 머신

```
            ┌──────────────────────────────────────────┐
            │           Strategy Lifecycle              │
            │                                          │
            │  CANDIDATE ──────▶ SHADOW ──────▶ LIVE   │
            │      ▲                              │    │
            │      │                              │    │
            │      └──── DEPRECATED ◀─────────────┘    │
            │                                          │
            └──────────────────────────────────────────┘

유효한 전이:
  CANDIDATE → SHADOW   (AST 검증 통과, 그림자 테스트 시작)
  SHADOW    → LIVE     (승격 조건 충족: Sharpe≥1.0, WR≥50%, N일 연속)
  SHADOW    → DEPRECATED (강등 조건: WR<40% or MDD>25%, M일 연속)
  LIVE      → DEPRECATED (성과 악화)
  DEPRECATED → CANDIDATE (재활성화)
```

### 14.2.2. StrategyEntry 데이터 구조

```python
@dataclass
class StrategyEntry:
    name: str                          # 고유 이름
    strategy_fn: Callable | None       # 실행 가능한 전략 함수
    status: StrategyStatus             # CANDIDATE/SHADOW/LIVE/DEPRECATED
    parameters: dict                   # 전략 파라미터
    metrics: dict                      # 성과 메트릭 (win_rate, sharpe, mdd)
    wfo_results: dict                  # WFO 최적화 결과
    source: str                        # "builtin" | "llm_generated"
    shadow_start: datetime | None      # SHADOW 전환 시점
```

### 14.2.3. 빌트인 전략 (5종)

| 전략 | 팩토리 함수 | WFO 파라미터 그리드 |
|------|------------|---------------------|
| **평균 회귀** | `create_mean_reversion_strategy()` | rsi_oversold: [25,30,35], rsi_overbought: [65,70,75] |
| **추세 추종** | `create_trend_following_strategy()` | ma_short: [10,20,30], ma_long: [40,50,60] |
| **모멘텀** | `create_momentum_strategy()` | roc_period: [5,10,14], roc_threshold: [2,3,5] |
| **브레이크아웃** | `create_breakout_strategy()` | lookback: [14,20,30], atr_multiplier: [1.5,2.0,2.5] |
| **복합** | `create_combined_strategy()` | rsi_oversold: [25,30], ma_short: [10,20] |

## 14.3. Walk-Forward Optimizer (WFO) — 파라미터 자동 최적화

과적합(Overfitting)을 방지하는 롤링 윈도우 기반 파라미터 최적화.

### 14.3.1. 동작 원리

```
전체 데이터
├──────────────────────────────────────────────────────┤
│                                                      │
│  Window 1:                                           │
│  ┌────── IS ──────┐┌── OOS ──┐                      │
│  │  in-sample     ││out-sample│                      │
│  │  Grid Search   ││  검증    │                      │
│  └────────────────┘└─────────┘                      │
│                                                      │
│  Window 2:                                           │
│       ┌────── IS ──────┐┌── OOS ──┐                 │
│       │  in-sample     ││out-sample│                 │
│       │  Grid Search   ││  검증    │                 │
│       └────────────────┘└─────────┘                 │
│                                                      │
│  Window 3:                                           │
│            ┌────── IS ──────┐┌── OOS ──┐            │
│            └────────────────┘└─────────┘            │

IS: 파라미터 최적화 구간 (기본 1000 bars)
OOS: 미래 데이터 검증 구간 (기본 336 bars = 2주)
Step: 윈도우 이동 간격 (기본 168 bars = 1주)
```

### 14.3.2. 과적합 탐지

| 메트릭 | 계산 | 판정 |
|--------|------|------|
| **OOS/IS Ratio** | OOS 스코어 / IS 스코어 | < 0.60 → 과적합 |
| **Overfit Ratio** | 과적합 윈도우 수 / 전체 윈도우 수 | > 0.50 → 전략 비신뢰 |
| **Objective** | `sharpe_ratio - 0.5 × (max_drawdown_pct / 100)` | 수익과 리스크 균형 |

### 14.3.3. 실행 스케줄

- **주기**: 매주 일요일 UTC 00:00
- **대상**: 모든 LIVE 전략
- **결과**: `is_robust=True`인 경우에만 파라미터 업데이트 (in-place)

## 14.4. Shadow Tester — 그림자 테스트

SHADOW 전략을 실자금 투입 없이 가상으로 실행하여 성과를 측정하는 시스템.

### 14.4.1. 승격/강등 조건

**승격 (SHADOW → LIVE)**:

| 조건 | 기본값 | 설명 |
|------|--------|------|
| Sharpe Ratio ≥ | 1.0 | 리스크 대비 수익 |
| 연속 일수 ≥ | 3일 | 일관된 성과 |
| Win Rate ≥ | 50% | 승률 |

**강등 (SHADOW/LIVE → DEPRECATED)**:

| 조건 | 기본값 | 설명 |
|------|--------|------|
| Win Rate < | 40% | 낮은 승률 |
| MDD > | 25% | 과도한 낙폭 |
| 연속 일수 ≥ | 2일 | 지속적 악화 |

### 14.4.2. QA 에이전트 통합

QA 에이전트가 그림자 테스트 관리를 담당한다:

```
[매일 evaluation_hour UTC]
QA Agent → ShadowTester.evaluate_daily()
  ├── 승격 대상 발견 → quant:strategy_promoted 이벤트 발행
  ├── 강등 대상 발견 → quant:strategy_demoted 이벤트 발행
  └── QA 리포트에 shadow_test 섹션 추가
```

## 14.5. Regime Classifier — 시장 국면 분류

현재 시장 상태를 3가지 국면으로 분류하여 전략별 가중치를 동적으로 조정한다.

### 14.5.1. 분류 규칙 (Phase 1: 규칙 기반)

```
ADX ≥ 25 → TRENDING   (confidence = min(adx/50, 1.0))
ATR percentile ≥ 75th → VOLATILE  (ATR lookback=100)
나머지 → RANGING       (confidence = 0.7)
```

### 14.5.2. 국면별 전략 가중치

| 국면 | trend_following | mean_reversion | momentum | breakout |
|------|----------------|----------------|----------|----------|
| **TRENDING** | 1.5 | 0.5 | 1.2 | 0.8 |
| **RANGING** | 0.5 | 1.5 | 0.8 | 0.7 |
| **VOLATILE** | 0.8 | 0.8 | 0.6 | 1.4 |

포트폴리오 관리 에이전트가 `quant:regime_classified` 이벤트를 구독하여
전략 가중치를 동적으로 조정한다.

### 14.5.3. 향후 확장 (Phase 2: HMM 기반)

`hmmlearn`, `scikit-learn`, `scipy`를 optional dependency로 추가하여
Hidden Markov Model 기반 확률적 국면 분류로 업그레이드 가능.
`[project.optional-dependencies] ml = [...]`로 관리.

## 14.6. Strategy Generator — LLM 전략 생성 (opt-in)

LLM을 활용하여 시장 상황에 맞는 새로운 전략 코드를 동적으로 생성한다.

> **보안 경고**: `exec()` 사용으로 인한 보안 리스크 존재.
> `evolution.generation_enabled = False` 기본값. 명시적 opt-in만 허용.

### 14.6.1. 보안 다층 방어

```
[Layer 1] AST 기반 정적 분석
  ├── BANNED_IMPORTS (28종): os, sys, subprocess, socket, http, ctypes, pickle...
  ├── BANNED_BUILTINS (16종): exec, eval, __import__, compile, open, getattr...
  ├── BANNED_ATTRIBUTES (11종): __subclasses__, __globals__, __builtins__...
  └── Class 정의 금지

[Layer 2] 정규식 보완 검사
  ├── 문자열 결합을 통한 AST 우회 패턴 탐지
  └── __import__, __subclasses__ 등 문자열 내 패턴

[Layer 3] 제한된 실행 환경
  ├── __builtins__: abs, min, max, sum, len, range, round, sorted, zip, enumerate, isinstance, float, int, str, list, dict, tuple, set, bool, print, True, False, None만 허용
  └── 허용 모듈: math, statistics (numpy, pandas는 설치 시 추가)

[Layer 4] 기능 검증
  └── 생성된 함수 시그니처/반환값 형식 검증
```

### 14.6.2. 생성 플로우

```
[QuantAgent] 저변동성 시간대 감지
    │
    ▼
quant:strategy_generate_request 발행
    │
    ▼
[SWEngineerAgent] StrategyGenerator.generate()
    ├── LLM에 전략 코드 생성 요청 (시장 컨텍스트 + 기존 성과 포함)
    ├── 응답에서 코드 블록 추출
    ├── CodeSafetyChecker.check() — 보안 검증
    │   ├── 통과 → 제한된 namespace에서 exec()
    │   └── 실패 → developer:strategy_rejected 발행
    ├── 함수 시그니처/반환값 검증
    └── CANDIDATE 상태로 Registry 등록
        └── developer:strategy_generated 발행
```
