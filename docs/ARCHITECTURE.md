# P.R.O.F.I.T.
## Predictive Routing & Orchestration Framework for Intelligent Trading

---

# 1. 시스템 개요

P.R.O.F.I.T.은 암호화폐 현물 자동매매를 위한 멀티 에이전트 시스템이다.
금융 공학 팀의 실제 조직 구조를 모델링하여, 각 에이전트가 전문 영역의 업무를 수행하고
계층적 합의 메커니즘을 통해 의사결정의 신뢰성을 확보한다.

### 핵심 설계 원칙
- **계층적 상향식(Hierarchical Bottom-Up) 아키텍처**: 논문(arXiv:2602.23330)의 3-Level 구조를 차용
- **이중화(Redundancy)**: 단일 에이전트 오판 방지를 위한 교차 검증 체계
- **현물 거래 전용**: 선물/마진 없이 현물 매매만 수행
- **24/7 무중단 운영**: OpenClaw + Docker 기반 서버 배포
- **관리자 인터페이스**: OpenClaw 메시징 + Web UI를 통한 설정/모니터링

# 3. 에이전트 상세 역할 정의

## 3.1. 오케스트레이터 (Orchestrator)
> 금융팀의 CTO / 팀 리더 역할

**핵심 책임**: 전체 워크플로우 조율, 에이전트 간 합의 도출, 최종 의사결정 승인

| 업무 | 설명 |
|------|------|
| 워크플로우 관리 | 매매 사이클(분석→전략→검증→실행)의 파이프라인 조율 |
| 합의 조율 | 에이전트 간 산출물의 논리적 정합성 검증 (코사인 유사도 측정) |
| 충돌 해소 | 에이전트 간 의견 불일치 시 가중 투표(Weighted Voting)로 최종 판단 |
| 비상 제어 | 시스템 이상 감지 시 전체 매매 중단(Global Circuit Breaker) |
| OpenClaw 연동 | 관리자 메시지 수신 → 적절한 에이전트로 라우팅 → 응답 취합 |

**왜 필요한가**: 단일 에이전트 오판 방지. 퀀트 에이전트가 매수 시그널을 보내도,
경제 분석 에이전트의 거시 평가와 리스크 관리 에이전트의 승인이 없으면 실행되지 않는다.

## 3.2. 경제 분석 에이전트 (Macro/Micro Analyst Agent)
> 금융팀의 리서치 애널리스트 역할

**핵심 책임**: 시장 환경 분석, 코인별 펀더멘탈 평가, 투자 유니버스 관리

### 3.2.1. 거시(Macro) 분석
| 데이터 소스 | 수집 항목 | 분석 목적 |
|-------------|-----------|-----------|
| 미 연준(Fed) | 금리 결정, FOMC 의사록 | 글로벌 유동성 방향 |
| CPI/PPI 지표 | 물가 상승률 추이 | 인플레이션 환경 판단 |
| DXY (달러 인덱스) | 달러 강세/약세 | 위험자산 선호도 |
| Fear & Greed Index | 시장 심리 지수 | 극단적 공포/탐욕 감지 |
| BTC Dominance | 비트코인 시장 점유율 | 알트코인 시즌 판단 |
| 총 시가총액 | 암호화폐 전체 시총 | 시장 규모 추이 |
| 스테이블코인 유입량 | USDT/USDC 발행량 변화 | 시장 자금 유입 판단 |

### 3.2.2. 미시(Micro) 분석 - 코인별
| 데이터 소스 | 수집 항목 | 분석 목적 |
|-------------|-----------|-----------|
| CoinGecko/CMC API | 시가총액, 거래량, 유통량 | 기본 지표 |
| 토큰 언락 일정 | 물량 해제 날짜, 해제 비율 | 매도 압력 예측 |
| 온체인 데이터 | 대규모 지갑 이동, DEX 거래량 | 스마트 머니 추적 |
| GitHub 활동 | 커밋 수, 개발자 수 | 프로젝트 활성도 |
| SNS/뉴스 | 트위터, 레딧, 텔레그램 감성 | NLP 기반 Sentiment Score |
| 거래소 상장/폐지 | 상장 예정, 상폐 경고 | 유동성 변화 예측 |

### 3.2.3. 코인 선별 (Universe Screening)

코인 선별은 경제 분석 에이전트가 1차 필터링하고, 퀀트 에이전트가 2차 기술적 평가를 수행한다.

```
[1차 필터: 경제 분석 에이전트]
  │
  ├── 시가총액 기준: Top 100 (유동성 확보)
  ├── 일평균 거래량: 최소 $10M 이상
  ├── 거래소 상장: 타겟 거래소에 상장된 코인
  ├── 토큰 언락: 30일 이내 대규모(유통량 5%+) 언락 예정 코인 경고 플래그
  ├── 프로젝트 리스크: 해킹 이력, 규제 이슈, 팀 이탈 등
  └── 결과: 후보군 리스트 (Watchlist) + 코인별 펀더멘탈 스코어
         │
[2차 필터: 퀀트 에이전트]
  │
  ├── 기술적 지표 기반 진입 타이밍 평가
  ├── 변동성(Volatility) 분석
  ├── 상관계수(Correlation): BTC와의 상관성
  └── 결과: 최종 매매 대상 + 보유 기간 분류
```

### 3.2.4. 보유 기간 분류 기준

| 분류 | 기간 | 선별 기준 | 비중 |
|------|------|-----------|------|
| **단기(Swing)** | 1~7일 | 기술적 과매도, 급등 모멘텀, 이벤트 드리븐 | 전체의 20~30% |
| **중기(Position)** | 1~4주 | 추세 전환 확인, 온체인 축적 신호 | 전체의 40~50% |
| **장기(Core)** | 1개월+ | 높은 펀더멘탈 스코어, BTC/ETH 등 대형 코인 | 전체의 20~30% |

## 3.3. 퀀트 에이전트 (Quant/Algorithm Agent)
> 금융팀의 퀀트 트레이더 역할

**핵심 책임**: 수학적 모델 기반 매매 전략 수립, 시그널 생성, 백테스팅

### 3.3.1. 전략 엔진

| 전략 카테고리 | 사용 지표 | 적용 대상 |
|---------------|-----------|-----------|
| 추세 추종 (Trend Following) | MA(20/50/200), MACD, ADX | 중기/장기 보유 |
| 평균 회귀 (Mean Reversion) | RSI, 볼린저 밴드, Z-Score | 단기 스윙 |
| 모멘텀 (Momentum) | ROC, 거래량 급증 감지 | 단기 진입 |
| 브레이크아웃 (Breakout) | 채널 돌파, ATR 기반 변동성 | 추세 전환 시점 |

### 3.3.2. 시그널 생성 프로세스

```
[경제 분석 에이전트 산출물]          [시장 데이터 (실시간)]
        │                                   │
        ▼                                   ▼
┌───────────────────────────────────────────────┐
│              퀀트 에이전트 엔진                  │
│                                               │
│  1. 지표 연산 (MA, RSI, MACD, BB 등)           │
│  2. 멀티 타임프레임 분석 (1H, 4H, 1D)           │
│  3. 시그널 스코어링 (-100 ~ +100)              │
│  4. 보유 기간 결정 (단기/중기/장기)               │
│  5. 진입가/목표가/손절가 계산                     │
│  6. 포지션 사이징 제안                           │
└───────────────────┬───────────────────────────┘
                    │
                    ▼
         [매매 시그널 (Trade Signal)]
         - 코인: BTC/USDT
         - 방향: BUY
         - 스코어: +72
         - 진입가: $65,200
         - 목표가: $68,500 (+5.06%)
         - 손절가: $63,400 (-2.76%)
         - 보유 기간: 중기 (7~14일)
         - 포지션 비중: 전체 자금의 15%
```

### 3.3.3. 백테스팅 모듈

| 지표 | 설명 | 기준값 |
|------|------|--------|
| 총 수익률 | 전략의 누적 수익 | 벤치마크(BTC) 대비 초과 수익 |
| MDD | 최대 낙폭 | -20% 이하 시 전략 재검토 |
| Sharpe Ratio | 위험 대비 수익 | 1.0 이상 |
| 승률 (Win Rate) | 수익 거래 비율 | 50% 이상 |
| 손익비 (Risk/Reward) | 평균 수익 / 평균 손실 | 1.5 이상 |

### 3.3.4. 매매 트리거 발생 메커니즘

**매매 트리거는 퀀트 에이전트에서 발생**하며, 다음 두 가지 경로로 작동한다:

```
[경로 1: 정기 스캔 (Scheduled Scan)]
  Cron Job (15분/1시간/4시간 주기)
    → 퀀트 에이전트: 전체 Watchlist 지표 연산
    → 임계값 돌파 코인 감지
    → 시그널 생성
    → 오케스트레이터로 전달

[경로 2: 실시간 이벤트 (Event-Driven)]
  WebSocket 가격 스트림
    → 급등/급락 감지 (5분 내 ±3% 이상)
    → 퀀트 에이전트: 긴급 시그널 평가
    → 시그널 생성
    → 오케스트레이터로 전달
```

## 3.4. 리스크 관리 에이전트 (Risk Manager Agent) ★신규★
> 금융팀의 리스크 매니저 역할

**핵심 책임**: 자금 관리, 손실 통제, 포트폴리오 건전성 유지

**왜 별도 에이전트인가**: 매매 결정을 내리는 에이전트와 리스크를 통제하는 에이전트가 같으면
이해 충돌이 발생한다. 독립적인 리스크 관리 에이전트는 "브레이크 역할"을 수행한다.

### 3.4.1. 예수금 관리

| 항목 | 규칙 | 설명 |
|------|------|------|
| **최소 보유금 (Reserve)** | 전체 자산의 30% | 어떤 상황에서도 투자에 사용하지 않는 안전 자금 |
| **가용 투자금** | 전체 자산의 70% | 실제 매매에 사용 가능한 자금 |
| **단일 포지션 한도** | 가용 투자금의 20% | 하나의 코인에 집중 투자 방지 |
| **일일 최대 손실** | 전체 자산의 -3% | 도달 시 당일 신규 매매 중단 |
| **총 최대 손실** | 전체 자산의 -10% | 도달 시 전체 매매 중단, 관리자 알림 |

### 3.4.2. 현물 거래에서의 리스크 완화 전략

> **Q: 암호화폐 거래소에 헷징 수단이 있는가?**
>
> 현물 거래만 지원하는 환경에서는 전통적인 헷징(선물 숏, 옵션 풋) 수단이 제한된다.
> 따라서 다음과 같은 **현물 기반 리스크 완화 전략**을 적용한다:

| 전략 | 방법 | 설명 |
|------|------|------|
| **스테이블코인 비중 조절** | USDT/USDC 비중 확대 | 하락장 감지 시 스테이블코인 비중을 60~80%까지 확대 |
| **분산 투자** | 상관계수 기반 분산 | BTC 상관성이 낮은 코인 편입으로 포트폴리오 분산 |
| **분할 매수/매도** | DCA(Dollar Cost Averaging) | 한 번에 전량 매매하지 않고 3~5회 분할 |
| **손절(Stop-Loss)** | 자동 손절 주문 | 코인별 최대 허용 손실률 도달 시 자동 매도 |
| **트레일링 스탑** | 수익 보호 | 고점 대비 일정 비율 하락 시 자동 매도 |
| **현금화 기준** | 시장 공포 지수 | Fear & Greed ≤ 20 시 보유 포지션 축소 |

### 3.4.3. 리스크 관리 에이전트가 수행하는 헷징 의사결정 흐름

```
[시장 상태 평가]
  │
  ├── 경제 분석 에이전트 → 거시 환경 리포트 수신
  ├── 퀀트 에이전트 → 변동성/추세 지표 수신
  │
  ▼
[리스크 스코어 산출] (0~100, 높을수록 위험)
  │
  ├── 0~30 (Low Risk): 가용 투자금 100% 활용 가능
  ├── 31~60 (Medium Risk): 가용 투자금 70% 활용, 스테이블코인 30%
  ├── 61~80 (High Risk): 가용 투자금 40%, 기존 포지션 축소 시작
  └── 81~100 (Critical): 신규 매매 중단, 기존 포지션 단계적 청산
         │
         ▼
[오케스트레이터에게 리스크 레벨 전달]
  → 매매 실행 에이전트의 주문 한도 조정
  → 포트폴리오 관리 에이전트에 리밸런싱 요청
```

# 4. 단일 에이전트 오판 방지: 교차 검증 및 합의 메커니즘

## 4.1. 위험 분석

> **Q: 각 에이전트가 한 개씩만 존재했을 때의 위험은?**

| 위험 시나리오 | 발생 가능 상황 | 결과 |
|---------------|----------------|------|
| 경제 분석 오판 | NLP 감성 분석 오류로 긍정 편향 | 하락장에서 매수 시그널 |
| 퀀트 오판 | 과최적화된 전략, 지표 오계산 | 잘못된 진입/청산 시점 |
| 리스크 관리 미작동 | 한도 초과 미감지 | 과도한 손실 |
| 실행 오류 | 주문 중복, 잘못된 수량 | 의도치 않은 대량 매매 |

## 4.2. 다층 방어 체계 (Defense in Depth)

```
[시그널 생성]     [독립 검증]       [합의]          [실행 전 최종 검증]
     │                │              │                    │
  퀀트 Agent    경제분석 Agent    Orchestrator      Risk Manager
  시그널 생성    시장 방향성 확인   두 산출물 대조     한도/규칙 준수 확인
     │                │              │                    │
     └────────────────┴──────────────┴────────────────────┘
                                     │
                            [모두 승인 시에만 실행]
                                     │
                              Execution Agent
```

### 4.2.1. 합의 프로토콜 (Consensus Protocol)

논문(arXiv:2602.23330)의 방법론을 적용한 4단계 합의:

**Step 1 - 전략 제안 (Proposal)**
- 퀀트 에이전트가 매매 시그널을 생성하여 오케스트레이터에 제출
- 시그널에는 수치적 근거(지표값, 스코어)가 포함

**Step 2 - 독립 검증 (Independent Validation)**
- 경제 분석 에이전트: 현재 거시/미시 환경이 시그널 방향과 일치하는지 평가
- 리스크 관리 에이전트: 포지션 한도, 가용 자금, 리스크 레벨 준수 여부 확인
- 두 에이전트는 서로의 판단에 영향받지 않도록 **독립적으로** 평가

**Step 3 - 정합성 검증 (Reconciliation)**
- 오케스트레이터가 세 에이전트의 산출물을 수집
- 퀀트 시그널의 방향성 vs 경제 분석의 시장 평가 간 **코사인 유사도** 측정
- 유사도 임계값(예: 0.6 이상) 미달 시 시그널 기각 또는 스케일 다운

**Step 4 - 최종 승인 및 실행 (Approval & Execution)**
- 3개 에이전트 중 2개 이상이 승인해야 실행 (2-out-of-3 Quorum)
- 리스크 관리 에이전트는 **거부권(Veto)** 보유: 리스크 기준 위반 시 단독 기각 가능
- 승인된 매매 지시는 포트폴리오 관리 에이전트를 통해 매매 실행 에이전트로 전달

### 4.2.2. 에이전트 간 견제 구조

```
┌─────────────────────────────────────────────────┐
│              견제 및 균형 구조                      │
│                                                  │
│  퀀트 Agent ←── 검증 ──→ QA Agent               │
│       │                      │                   │
│       │ 시그널               │ 로직 검증          │
│       ▼                      ▼                   │
│  Orchestrator ◄── 거부권 ── Risk Manager         │
│       │                      │                   │
│       │ 승인                 │ 한도 확인          │
│       ▼                      ▼                   │
│  Portfolio Manager ──────→ Execution Agent       │
│                                                  │
│  경제분석 Agent: 독립적 시장 평가로 교차 검증       │
└─────────────────────────────────────────────────┘
```

### 6.1.2. 투자 견해 질의 처리 흐름

# 12. 핵심 Q&A 요약

| 질문 | 답변 |
|------|------|
| 단일 에이전트 오판 위험? | 2-out-of-3 Quorum + 리스크 에이전트 거부권으로 방지 |
| 코인 선별은 어떻게? | 경제분석(1차 펀더멘탈) → 퀀트(2차 기술적) 2단계 필터링 |
| 보유 기간 결정은? | 퀀트 제안 → 포트폴리오 매니저 최종 결정 |
| 헷징 전략은 누가? | 리스크 관리 에이전트 (현물 기반: 스테이블코인 비중 조절 + 분산) |

## 13.2. P.R.O.F.I.T.의 강점 (업계 대비 우수한 점)

1. **멀티 에이전트 합의 구조**: 단일 봇 시스템 대비 의사결정 견고성이 우수. 2-out-of-3 쿼럼 + 리스크 거부권은 학술 근거(arXiv:2602.23330)가 있는 설계
2. **펀더멘탈+기술적 2단계 선별**: 상용 봇들이 기술적 지표만 사용하는 것과 달리, 거시/미시 경제 분석을 통합한 코인 선별
3. **리스크 관리 독립성**: 매매와 리스크의 분리는 기관급 설계 원칙을 정확히 반영
4. **LLM 네이티브 아키텍처**: Claude API 기반 자연어 분석/판단/인터페이스 통합은 기존 퀀트 플랫폼에 없는 차세대 접근
5. **투명한 의사결정 추적**: 에이전트 판단 근거, 합의 과정, 코사인 유사도까지 기록하는 매매일지

## 13.3. 식별된 핵심 갭 및 보완 사항

### 13.3.1. [P1] OMS 상태 동기화 + 멱등성 키 (Idempotency Keys)

> **위험**: 네트워크 단절 시 "좀비 주문(Zombie Order)"으로 인한 중복 주문 → 직접적 재정 손실

**현행 설계의 문제**:
매매 실행 에이전트가 주문 전송과 체결 확인에 초점이 맞춰져 있으나,
주문의 전체 생명주기(생성→전송→부분체결→완전체결→취소→수정)를 관리하는
상태 머신(Order State Machine)이 부재하다.

**보완 설계**:

```
[주문 상태 머신 (Order State Machine)]

  CREATED ──→ SUBMITTED ──→ PARTIALLY_FILLED ──→ FILLED
     │            │                │                  │
     │            ▼                ▼                  ▼
     │        REJECTED         CANCELLED          [완료]
     │            │                │
     ▼            ▼                ▼
  [에러 처리]   [로그 기록]      [잔여 수량 처리]
```

| 구현 항목 | 설명 |
|-----------|------|
| **멱등성 키(Idempotency Key)** | 각 주문에 UUID 기반 고유 키 부여. Binance `newClientOrderId`, Coinbase `client_order_id` 활용 |
| **Redis 키-상태 매핑** | 멱등성 키와 주문 상태를 Redis에 저장 (TTL: 주문 유효기간 + 여유) |
| **중복 주문 방지** | 동일 키로 재전송 시 기존 결과 반환, 새 주문 생성하지 않음 |
| **주기적 조정(Reconciliation)** | 거래소 실제 주문 상태와 내부 OMS 상태를 5분 주기로 동기화 |
| **감사 추적(Audit Trail)** | 모든 주문 이벤트(생성, 수정, 취소, 체결, 오류)를 불변 로그로 기록 |

**Execution Agent 명세 확장**:

| 기능 (기존) | 추가 기능 |
|------------|-----------|
| 주문 라우팅 | + 멱등성 키 생성 및 관리 |
| 체결 확인 | + 주문 상태 머신 전이 관리 |
| 슬리피지 모니터링 | + 거래소-내부 상태 조정(Reconciliation) |
| 서킷 브레이커 | + 좀비 주문 감지 및 자동 복구 |
| 매매 로그 기록 | + 완전한 주문 감사 추적(Audit Trail) |

### 13.3.2. [P2] 백테스트-라이브 통합 이벤트 엔진

> **위험**: 백테스팅 코드와 라이브 실행 코드가 분리되면 "백테스트에서만 잘 되는" 전략이 생산됨

**현행 설계의 문제**:
백테스팅(Backtrader/vectorbt)과 라이브 실행이 별도 코드 경로로 설계되어 있어,
Look-Ahead Bias, Survivorship Bias 위험이 존재한다.

**보완 설계: 통합 이벤트 엔진 (Unified Event Engine)**

```
┌──────────────────────────────────────────────────┐
│            Unified Event Engine                   │
│                                                   │
│  ┌──────────────┐     ┌──────────────┐           │
│  │  Historical   │     │  Live        │           │
│  │  Data Feed    │     │  Data Feed   │           │
│  │  (백테스트용)  │     │  (실시간용)   │           │
│  └──────┬───────┘     └──────┬───────┘           │
│         │                     │                   │
│         └──────────┬──────────┘                   │
│                    ▼                              │
│         ┌──────────────────┐                      │
│         │  Event Dispatcher │  ← 동일 인터페이스   │
│         │  on_tick()        │                      │
│         │  on_bar()         │                      │
│         │  on_order()       │                      │
│         │  on_fill()        │                      │
│         └────────┬─────────┘                      │
│                  ▼                                │
│         ┌──────────────────┐                      │
│         │  Strategy Code   │  ← 백테스트/라이브    │
│         │  (동일 코드)      │     동일 코드 실행    │
│         └────────┬─────────┘                      │
│                  ▼                                │
│         ┌──────────────────┐                      │
│         │  Execution Layer │                      │
│         │  - Simulated     │  ← 백테스트/Paper    │
│         │  - Live          │  ← 실제 거래소       │
│         └──────────────────┘                      │
└──────────────────────────────────────────────────┘
```

| 구현 항목 | 설명 |
|-----------|------|
| **추상 DataFeed** | Historical/Live 모두 동일 인터페이스(on_tick, on_bar)로 전략에 데이터 전달 |
| **추상 Broker** | SimulatedBroker(백테스트/Paper) / LiveBroker(실제 거래소)로 체결 분기 |
| **현실적 백테스트** | 슬리피지 모델(Volume-based), 커미션 모델, 시장 영향(Market Impact) 모델 포함 |
| **Walk-Forward Analysis** | 과최적화 방지를 위한 Walk-Forward Optimization, Monte Carlo Simulation |

### 13.3.3. [P3] CI/CD + Sandbox Forward Testing (Paper Trading)

> **위험**: 새로운 전략/코드를 프로덕션에 직접 배포하면 버그로 인한 즉각적 재정 손실 발생

**보완 설계: 4단계 전략 배포 파이프라인**

```
Stage 1: 검증 (CI)
──────────────────
  코드 변경 → Lint + Unit Test + Integration Test
  ├── 통과 → Stage 2로
  └── 실패 → QA 에이전트 피드백 → 재작업

Stage 2: 백테스트
──────────────────
  과거 6개월 데이터로 자동 백테스트
  ├── 성과 기준 충족 확인:
  │     Sharpe > 1.0
  │     MDD < -20%
  │     Win Rate > 50%
  │     손익비 > 1.5
  ├── 통과 → Stage 3로
  └── 미달 → 전략 재검토

Stage 3: Paper Trading (Forward Test)
──────────────────────────────────────
  Sandbox 컨테이너에서 실시간 데이터 + 가상 체결
  ├── 최소 2주간 운영
  ├── 실제 시장 데이터 수신, 가상 체결 엔진으로 주문 처리
  ├── 성과 리포트 자동 생성
  ├── 통과 → Stage 4로
  └── 미달 → Stage 2로 회귀

Stage 4: 프로덕션 배포
──────────────────────
  관리자 수동 승인 → 소액(전체 자금 10%) 실거래 시작
  ├── 1주간 소액 운영
  ├── 성과 확인 후 전체 자금 적용
  └── 이상 시 자동 롤백
```

**Docker 환경 추가 서비스**:

| 서비스 | 역할 |
|--------|------|
| `profit-sandbox` | Paper Trading 전용 컨테이너. 실시간 데이터 수신 + 가상 체결 |
| `profit-ci` | 코드 변경 시 자동 테스트/백테스트 실행 |

### 13.3.4. [P4] 에이전트 성과 추적 및 동적 가중치

> Numerai의 메타모델처럼, 에이전트별 과거 정확도를 기반으로 합의 가중치를 동적 조정

**현행 설계의 문제**:
"가중 투표(Weighted Voting)"가 명시되어 있으나, 가중치 산출 방법이 불명확하다.

**보완 설계**:

| 추적 지표 | 설명 | 가중치 반영 |
|-----------|------|-------------|
| 시그널 정확도 | 에이전트 예측 방향 vs 실제 가격 변동 | 최근 30일 이동 정확도 |
| 리포트 신뢰도 | 분석 리포트의 결론 vs 실제 시장 방향 | EMA(지수이동평균) 기반 |
| 리스크 판단 정확도 | 리스크 경고 발동 vs 실제 손실 발생 | 적중률 기반 |

```
[에이전트 성과 추적 흐름]

  매매 완료 → 실제 결과 기록
       │
       ▼
  각 에이전트의 과거 판단과 대조
       │
       ├── 경제 분석: 시장 방향 예측 정확도
       ├── 퀀트: 시그널 스코어 vs 실제 수익률 상관계수
       └── 리스크: 위험 경고의 적중률
       │
       ▼
  에이전트별 신뢰도 점수 갱신 (0.0 ~ 1.0)
       │
       ▼
  다음 합의 투표 시 가중치로 반영
  (신뢰도가 높은 에이전트의 의견에 더 높은 가중치)
```

### 13.3.5. [P5] 거래 비용 분석 (TCA) 모듈

> 기관 펀드 수준의 실행 품질 측정 및 개선

| **P4** | 에이전트 성과 추적 + 동적 가중치 | 높음 | 중간 | 합의 메커니즘 실효성 강화 |
| **P5** | TCA 모듈 | 중간 | 중간 | 기관급 실행 품질 |
| **P6** | Admin UI 분리 (React + Grafana) | 중간 | 낮음 | 개발 효율성 향상 |
| **P7** | 크리덴셜 보안 강화 | 중간 | 낮음~중간 | Docker Secrets 우선 적용 |
